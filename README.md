## Usage Instructions
1. Open Vivado and create a new RTL project from the Vivado Quick Start menu.
2. Download all .vhd source files from the GitHub repository and add them during the Add Sources step.
3. Download the doodle_jump.xdc constraints file from GitHub and add it during the Add Constraints step.
4. Connect the monitor to the Nexys board by plugging the monitor’s HDMI cable into an HDMI-to-VGA adapter. Then connect the VGA side of the adapter to the VGA port on the Nexys board.
5. Ensure the Nexys board is powered on and connected to the computer through its USB programming cable.
6. Run Synthesis.
7. Run Implementation.
8. Generate the Bitstream.
9. Open Hardware Manager, select Auto Connect, and program the Nexys board with the generated bistream.
10. Once programming is complete, Doodle Jump (1984) should appear on the connected monitor.

## Inputs and Outputs
### `fsm.vhd`
The `fsm` module serves as the top-level game controller.
- In state `S0`, the game is idle; FSM waits for a high `BTN0` input, which is converted into a one-cycle `start_pulse`
- In state `S1`, the game is running; `BTNL` and `BTNR` are passed as `move_left_i` and `move_right_i` to drive horizontal motion in `doodle_jump.vhd`; monitors `score_at_target`, `fall_complete`, and
- `pause_internal` to determine win, lose, or pause states
- In state `S2`, the user wins given the score returned from `doodle_jump.vhd` is greater than or equal to generic threshold `N`
- In state `S3`, the user loses when `fall_complete` is asserted, indicating that the doodler has fallen below the valid gameplay boundary
- In state `S4`, the game is paused via a high `pause_internal` signal, generated by a button-toggle process
- FSM outputs the current state through `Y_i` and maps state-specific signals to `doodle_jump.vhd`, driving correct gameplay behavior and appropriate routing to display the final visuals on VGA display and seven-segment display

```
ENTITY fsm IS
    GENERIC (
        N : INTEGER := 16#0100#
    );
    PORT (
        clk_in : IN STD_LOGIC;
        BTNL : IN STD_LOGIC;
        BTNR : IN STD_LOGIC;
        BTN0 : IN STD_LOGIC;
        BTNU : IN STD_LOGIC;
        BTND : IN STD_LOGIC;
        VGA_red : OUT STD_LOGIC_VECTOR(3 DOWNTO 0);
        VGA_green : OUT STD_LOGIC_VECTOR(3 DOWNTO 0);
        VGA_blue : OUT STD_LOGIC_VECTOR(3 DOWNTO 0);
        VGA_hsync : OUT STD_LOGIC;
        VGA_vsync : OUT STD_LOGIC;
        SEG7_anode : OUT STD_LOGIC_VECTOR(7 DOWNTO 0);
        SEG7_seg : OUT STD_LOGIC_VECTOR(6 DOWNTO 0)
    );
END fsm;
```

**Inputs**
- `clk_in` - serves as the primary clock signal, configured at 25 MHz to provide the timing reference for state transitions, button detection, gameplay updates, VGA timing, and display refresh behavior
- `BTNL` - moves doodler to the left; input is only passed to game logic when the FSM is in the running state
- `BTNR` - moves doodler right; input is only passed to game logic when the FSM is in the running state
- `BTN0` - starts the game from idle state; a rising-edge pulse is generated from this button so that a single press initiates gameplay without requiring the button to be held
- `BTNU` - toggles pause state; pressing once pauses gameplay, and pressing it again resumes
- `BTND` - resets the game; when asserted, the FSM returns to the idle state and the game logic resets the doodler position, score, platforms, and related gameplay signals

**Outputs**
- `VGA_red` - 4-bit red color output
- `VGA_green` - 4-bit green color output
- `VGA_blue` - 4-bit blue color output
- `VGA_hsync` - horizontal synchronization signal to coordinate pixel scanning across each row of the display
- `VGA_vsync` - vertical synchronization signal to coordinate frame refresh and movement from one frame to the next
- `SEG7_anode` - selects which digit of the multiplexed seven-segment display is currently active; enables displayable scoreboard across multiple digits
- `SEG7_seg` - determines which LED segments are illuminated; with `SE7_anode`, the output displays the player’s current score


### `doodle_jump.vhd`
The `doodle_jump` module serves as the low-level gameplay engine. It operates under control of `fsm.vhd`, performing the gameplay operations associated with those states.
- `doodle_jump.vhd` receives state-control signals `game_reset`, `game_running`, `game_idle_state`, `game_pause_state`, `game_won_state`, and `game_lost_state`
- When `game_running` is high, live modifications are made to doodler position, platform collision detection, score incrementation, platform panning, and platform generation; otherwise, gameplay mechanics are disabled
- High `game_reset` resets the internal game state: returns doodler to starting position, resets jump counter, restores initial platform positions, cleans broken-platform flags, and reinitializes LFSR seed
- RGB multiplexer selects final `VGA_red`, `VGA_green`, and `VGA_blue` outputs to be sent to the VGA driver
- Score is stored internally as `display` and sent to both `score_out` and the seven-segment display decoder; `score_out` allows `fsm.vhd` to check win condition while `SEG7_anode` and `SEG7_seg` displays the current score to user

```
ENTITY doodle_jump IS
    PORT (
        clk_in : IN STD_LOGIC;
        game_reset : IN STD_LOGIC;
        game_running : IN STD_LOGIC;
        move_left : IN STD_LOGIC;
        move_right : IN STD_LOGIC;
        state_y : IN STD_LOGIC_VECTOR(2 DOWNTO 0);
        game_idle_state : IN STD_LOGIC;
        game_pause_state : IN STD_LOGIC;
        game_won_state : IN STD_LOGIC;
        game_lost_state : IN STD_LOGIC;
        score_out : OUT STD_LOGIC_VECTOR(15 DOWNTO 0);
        fall_complete : OUT STD_LOGIC;
        VGA_red : OUT STD_LOGIC_VECTOR(3 DOWNTO 0);
        VGA_green : OUT STD_LOGIC_VECTOR(3 DOWNTO 0);
        VGA_blue : OUT STD_LOGIC_VECTOR(3 DOWNTO 0);
        VGA_hsync : OUT STD_LOGIC;
        VGA_vsync : OUT STD_LOGIC;
        SEG7_anode : OUT STD_LOGIC_VECTOR(7 DOWNTO 0);
        SEG7_seg : OUT STD_LOGIC_VECTOR(6 DOWNTO 0)
    );
END doodle_jump;
```

**Inputs**
- `clk_in` - primary clock input used to drive gameplay updates, score tracking, platform regeneration, and display-related timing
- `game_reset` - resets internal gameplay state
- `game_running` - enables active gameplay updates
- `move_left` - moves the doodler left; derived from `BTNL` in `fsm.vhd`
- `move_right` - moves the doodler right; derived from `BTNR` in `fsm.vhd`
- `state_y` - provides the current FSM state code
- `game_idle_state` - selects idle screen
- `game_pause_state` - selects pause screen and freezes gameplay updates
- `game_won_state` - selects win screen when the FSM determines that the score has reached the target value
- `game_lost_state` - selects lose screen when doodler has fallen below gameplay boundary

**Outputs**
- `score_out` - sends current score back to `fsm.vhd` to determine whether win condition has been met
- `fall_complete` - indicates that the doodler has fallen below the allowed threshold, allowing `fsm.vhd` to transition into lose state
- `VGA_red` - 4-bit red color output
- `VGA_green` - 4-bit green color output
- `VGA_blue` - 4-bit blue color output
- `VGA_hsync` - horizontal synchronization signal to coordinate pixel scanning across each row of the display
- `VGA_vsync` - vertical synchronization signal to coordinate frame refresh and movement from one frame to the next
- `SEG7_anode` - selects which digit of the multiplexed seven-segment display is currently active; enables displayable scoreboard across multiple digits
- `SEG7_seg` - determines which LED segments are illuminated; with `SE7_anode`, the output displays the player’s current score


### `doodler.vhd`
The `doodler` module draws the doodler avatar at its current x and y position on the VGA display, providing the characteristics for its head, eyes, body, and feet.
- Receives the doodler’s current position from `doodle_jump.vhd` and determines whether the current VGA pixel should be part of the doodler sprite
- Uses `pixel_row` and `pixel_col` from VGA timing system to compare the current pixel location against several rectangular regions that form the avatar
- Doodler’s avatar is drawn centered around `doodler_x` and `doodler_y` at coordinates (400, 300) respectively
- Outputs the doodler’s feet position via `feet`, which is used by `doodle_jump.vhd` for platform collision detection and jump reset behavior

```
ENTITY doodler IS
    PORT (
        v_sync : IN STD_LOGIC;
        pixel_row : IN STD_LOGIC_VECTOR(10 DOWNTO 0);
        pixel_col : IN STD_LOGIC_VECTOR(10 DOWNTO 0);
        doodler_x : IN STD_LOGIC_VECTOR (10 DOWNTO 0); -- current doodler x position
        doodler_y : IN STD_LOGIC_VECTOR (10 DOWNTO 0); -- current doodler y position
        feet : OUT STD_LOGIC_VECTOR (10 DOWNTO 0);     -- current doodler feet position (for jump logic)
        red : OUT STD_LOGIC;
        green : OUT STD_LOGIC;
        blue : OUT STD_LOGIC
    );
END doodler;
```

**Inputs**
- `pixel_row` - current vertical pixel coordinate from VGA timing logic; compared against doodler’s y position
- `pixel_col` - current horizontal pixel coordinate from VGA timing logic; compared against doodler’s x position
- `doodler_x` - horizontal center position of the doodler provided by `doodle_jump.vhd` which updates the position based on user movement inputs and screen-wrapping logic
- `doodler_y` - vertical reference position of the doodler provided by `doodle_jump.vhd`, which updates according to the jump cycle, falling behavior, and platform landing detection

**Outputs**
- `feet` - outputs current vertical foot position to determine platform contact
- `red` - red output for doodler sprite; not used, so remains low
- `green` - green output for doodler sprite; asserted when `doodler_on` is high, causing the body, head, and feet of the avatar to appear green
- `blue` - blue output for doodler sprite; asserted when either eye region is active


### `platform.vhd`
The `platform` module draws the individual platform on the VGA display and applies behavior based on the platform’s assigned type.
- Platform types are determined by the 2-bit `platform_type` input
- `platform_type = “01”` produces moving blue platforms that move horizontally within a fixed range
- `platform_type = “10”` produces breaking yellow platforms that disappear when its corresponding `broken` signal is high
- Remaining bit patterns produce default green platforms that remain stationary
- Platform is drawn as a narrow rectangle using constants `platform_w` and `platform_h`; the platform has a 20-pixel half-width and a 4-pixel half-height
- The `update_platform` process updates the platform’s internal position once per `v_sync` rising edge, storing `spawn_x`, `spawn_y`, and `spawn_type` from `doodle_jump.vhd` then resets the platform’s current position to the new spawn location
- For moving blue platforms, `curr_x` updates by `move_speed` pixels each frame until it reaches `spawn_x + move_range`, then reverses direction
- The module outputs `curr_platform_pos_x` and `curr_platform_pos_y` so that `doodle_jump.vhd` can use the platform’s actual displayed position for collision detection; this is especially important for blue platforms because their position changes over time and is almost always different from the original spawn location
- The `create_platform` process handles VGA pixel drawing; if `broken` is low and the current pixel lies inside platform bounds, the internal `platform_on` signal is high

```
ENTITY platform IS
    PORT (
        clk_in : IN STD_LOGIC;
        v_sync : IN STD_LOGIC;
        pixel_row : IN STD_LOGIC_VECTOR(10 DOWNTO 0);
        pixel_col : IN STD_LOGIC_VECTOR(10 DOWNTO 0);
        platform_pos_x : IN STD_LOGIC_VECTOR (10 DOWNTO 0);
        platform_pos_y : IN STD_LOGIC_VECTOR (10 DOWNTO 0);
        platform_type : IN STD_LOGIC_VECTOR (1 DOWNTO 0);
        broken : IN std_logic;
        curr_platform_pos_x : OUT STD_LOGIC_VECTOR(10 DOWNTO 0);
        curr_platform_pos_y : OUT STD_LOGIC_VECTOR(10 DOWNTO 0);
        red : OUT STD_LOGIC;
        green : OUT STD_LOGIC;
        blue : OUT STD_LOGIC   
    );
END platform;
```

**Inputs**
- `v_sync` - vertical sync input used to update platform movement once per frame
- `pixel_row` - current vertical pixel coordinate from VGA timing logic; compared against platform’s y position
- `pixel_col` - current horizontal pixel coordinate from VGA timing logic; compared against platform’s x position
- `platform_pos_x` - spawn or assigned x position received from `doodle_jump.vhd`
- `platform_pos_y` - spawn or assigned y position received from `doodle_jump.vhd`
- `platform_type` - 2-bit input defining platform behavior and color
- `broken` - indicates whether the platform should disappear; when `broken = ‘1’`, the platform is not drawn, which is used for breaking yellow platforms after collision

**Outputs**
- `curr_platform_pos_x` - outputs platform’s current x position; allows `doodle_jump.vhd` to perform collision detection
- `curr_platform_pos_y` - outputs platform’s current y position; allows `doodle_jump.vhd` to perform collision detection
- `red` - red output for the platform; asserted with `green` for breaking yellow platforms
- `green` - green output for the platform; asserted for default green platforms and with `red` for breaking yellow platforms
- `blue` - blue output for the platform; asserted for moving blue platforms

**ChatGPT was used to streamline the development of coordinate-based conditionals for the instructions text, “PRESS BTN0 TO START JUMPING” and “PRESS BTND TO RESET”.**

## Modifications
Developing Doodle Jump (1984) began by integrating supporting modules from _Pong_ (Lab 6). `clk_wiz_0.vhd`, `clk_wiz_0_clk_wiz.vhd`, `leddec16.vhd`, and `vga_sync.vhd` were reused as foundational components for clock generation, seven-segment display control, and VGA signal timing. Minor modifications were made to `leddec16.vhd` so that all anodes addressed by the 3-bit digit selector could support the updating scoreboard. The remaining files were left unchanged.

The reused VGA synchronization structure was essential for rendering visible game elements. Notably, the `VGA_vsync` and `VGA_hsync` signals - along with the generated pixel row and column coordinates - provided the basis for drawing the doodler, platforms, and FSM-controlled screen displays. Since _Pong_ already demonstrated how color-based objects could be mapped to a VGA monitor, it provided a reliable starting point.

After establishing the foundation, further game logic development proceeded with furnishing four new key mechanics.

## New Implementations
### Moore Finite State Machine
A five-state FSM (Idle, Play, Pause, Win, Lose) coordinates the overall game flow. Idle holds all gameplay logic frozen until the user starts a session. Play runs the main loop while monitoring the landing pulse and fall-complete signal from the collision module. Reaching the target score transitions the machine to Win, and a completed fall transitions it to Lose. Both Win and Lose return to Idle on reset. Pause provides a clean suspend/resume path without disturbing the player position or platform module. User inputs on BTN0, BTNU, and BTND drive the remaining transitions for start, pause, and reset, respectively. The FSM owns all start/restart authority and acts as the game-master.

- First, the screen shows a starting screen when the game first loads. 
- BTND is pressed to transition to S0 (Idle). 
   - Then, BTN0 should be pressed to transition to S1 (Play); BTND could also be pressed while we are in S1 (Play) to transition back to S0 (Idle). 
- While we are in S1 (Play), game logic is mostly implemented here. 
   - The player can press BTNU to pause the game and transition to S4 (Pause) and vice versa to transition back to S1 (Play). 
- When at S1 (Play), if the player scores a cumulative of 256 points in decimal, they will transition to S2 (Win). 
   - The screen will output: “YOU WIN” and the player can press BTND to transition back to S0 (Idle). 
- Similarly, when at S1 (Play), if the player falls the FSM will register Fall = “1”. Then, they will transition to S3 (Lose). 
   - The screen will output: “YOU LOSE” and the player can press BTND to transition back to S0 (Idle).

### FSM-Based Screen Assignments
The display pipeline routes one of four pixel sources to the VGA monitor based on the current game state.

 - Screen modules: Four parallel RGB generators run concurrently, each producing pixel-level red/green/blue outputs indexed by pixel_row / pixel_col:

    - idle_screen: start/menu graphics
    - win_screen: victory display
    - lose_screen: game-over display
    - normal gameplay RGB: doodler sprite + platforms

 - Final RGB mux: A combinational selector driven by the FSM state outputs (game_idle_state, game_won_state, game_lost_state). Selection priority is idle → win → lose → gameplay, so terminal states override active rendering. Default (no flag asserted) falls through to gameplay. The mux output bus is red_in[3:0], green_in[3:0], blue_in[3:0]: 4-bit per channel matching the VGA DAC resistor ladder.

    - vga_sync: Consumes the muxed RGB and the pxl_clk from clk_wiz_0 to produce VGA-compliant timing: VGA_red/green/blue[3:0], VGA_hsync, VGA_vsync. It also generates the pixel_row / pixel_col counters fed back to the screen modules, and the v_sync tick used by gameplay logic for frame-rate updates.
    - Clock: clk_wiz_0 divides the 100 MHz board clock down to the 40 MHz pixel clock required for 800×600 @ 60 Hz.
